## 結果：問題リポ別の詳細

### 🔴 重大1: `happiness-system` — default branch が `claude/flourish-forge-setup-wpvm3n`

| 項目 | 状態 |
|---|---|
| default_branch | `claude/flourish-forge-setup-wpvm3n`（**main 不在**） |
| 作成日 | 2026-06-23 |
| 直近 push | 2026-06-24（活発） |
| ブランチ数 | 1 本のみ（その claude/* が default 化） |
| README | あり |
| CLAUDE.md | あり |

#### 中身を見ると
- リポ名は `happiness-system` だが、CLAUDE.md タイトルは「**flourish-forge — 運用憲法**」
- 独自設計：4エージェント逐次パイプライン、`/savor` `/scan` `/recall` `/morning` `/reflect` `/continue` のスラッシュコマンド構造、`session.json` 中心の状態管理
- CLAUDE.md は **50行以内** の独自憲法形式（他42リポとは設計思想が異なる）
- 「ブランチ作成禁止」と明記されているが、**main を作らず claude/* を default にしている** という変わった構成

#### 評価
これは「ガバナンス違反」ではなく **意図的な独自運用** の可能性が高い：
- 2日前に作って毎日活発に使われている → 現役プロジェクト
- 独自憲法形式が貫かれている → デザイン上の選択
- ただし `claude/flourish-forge-setup-wpvm3n` を default にする運用は、本リポの「main集約」原則と衝突する

**ユーザー判断が必要な分岐**:
- **A案**: 独自設計を尊重し、本リポ監査の例外リスト（`audits/EXCLUSIONS.md`）に登録して以後の自動監査から除外
- **B案**: claude/* ブランチ を `main` へリネーム（API 1コール）し、ガバナンス標準に揃える
- **C案**: 独自設計を維持しつつ、`main` を新規作成して並走させる（最も複雑）

branch-cleanup スキルの「② 取り込み不要なら削除」分岐の対象。

### 🟡 中2: 全43リポで「上位ガバナンス参照」リンク欠落（43/43）

#### 背景
- `claude-governance` リポ（本リポ）を作成したのは 2026-06-23
- その時点で「(B) いったん現状で運用開始し、次のグローバル更新時に同期フローが機能するか実地検証する」を Next Action として保留
- **当時の判断は「机上で書いた参照リンクは使われない可能性がある」だったが、現在の実態は**：
  - 各リポ CLAUDE.md には `branch-cleanup` トリガーや `Web版でも実行可能` 等の **本リポで定義した運用要素が既に反映されている**
  - つまり「本リポは既に正典として機能している」が、参照リンクが無いため、新セッションのClaude（特にWeb版）は **本リポを知らずに作業を始める**

#### 評価
このギャップは「保留判断の延長」だが、ガバナンス機能を最大化するなら追加すべき。
**今後グローバル更新があるたびに本リポへ来てもらう導線が無い** ことが、`claude-governance` の存在意義を半分以下にしている。

## 改善計画

### 優先順位の根拠

| 課題 | 影響 | 工数 | 緊急度 |
|---|---|---|---|
| (1) NASDAQ_backtest 余分ブランチ削除 | 小（identical なので失うもの無し） | 5分 | **高**（ガバナンス清浄化の即実行可） |
| (2) 上位ガバナンス参照リンク追加（43リポ） | 中（新セッションの導線確立） | 30-60分（並列サブエージェント） | **高**（ガバナンス機能の発揮） |
| (3) happiness-system の方針確定 | 中（独自運用 vs 標準化の選択） | ユーザー判断＋実行30分 | **中**（現役で使えているなら緊急ではない） |
| (4) 監査の自動化（routinize） | 大（ドリフト検知の継続実行） | 60分 | **中**（一度作れば毎月効く） |

### Phase 1: 即実行（ユーザー追加判断不要・低リスク）

#### 1-1. NASDAQ_backtest の余分ブランチ削除
- 確認済: main と identical（固有コミットゼロ）
- 削除手順: `git push origin --delete claude/sbi-click-365-nasdaq-guide-hfw748`
- 検証: 削除後 `git ls-remote --heads` で main のみ残ることを確認

### Phase 2: ユーザー判断後に実行

#### 2-1. 上位ガバナンス参照リンクを43リポへ一括追加
- 内容: 各リポ CLAUDE.md の末尾セクション「上位ガバナンスへの参照」を追加もしくは更新
- 文言（テンプレ §10 と同一）:
  ```markdown
  ## 上位ガバナンスへの参照
  - 本リポの運用は [KazuyaMurayama/claude-governance](https://github.com/KazuyaMurayama/claude-governance) の正典に準拠する
  - 競合した場合は本リポの CLAUDE.md が優先（リポ固有ルールが上位ガバナンスに勝つ）
  - 上位ガバナンスを変更した際は、本リポの CLAUDE.md にも反映する責務がある
  ```
- 実行方法: branch-cleanup の展開で実証済みの **6サブエージェント並列＋パイロット1** 方式
- マーカー: `<!-- GOVERNANCE_LINK_START/END -->` を新設（将来の一括更新を容易化）
- 検証: 監査スクリプトを再実行し GV 列が全て ✓ になることを確認

#### 2-2. happiness-system の方針確定
- 推奨: **A案（例外リスト登録）**
  - 理由: 2日前に作成され活発に使用中＝設計が「成功している」段階。今ガバナンス標準を強制すると独自設計の良さが壊れる
  - 実装: `audits/EXCLUSIONS.md` を新設し、リポ名と除外理由（独自憲法形式・main 不在設計）を明記
  - 監査スクリプトに除外ロジックを追加（`EXCLUSIONS.md` を読んで skip）

### Phase 3: 継続運用化（オプション・後日）

#### 3-1. 監査自動化
- 月次の `/schedule` ルーチン化、もしくは GitHub Actions スケジューラ
- 出力: `audits/AUDIT_YYYYMMDD.md` を本リポにコミット
- 変化点アラート: 前回比で「新たに不適合になった項目」を CHANGELOG.md にも自動追記
- これがあれば、グローバル更新→各リポ反映のドリフトを自動検知できる

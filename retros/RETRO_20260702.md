## 教訓（次セッションへ）

### 1. 「不適合ゼロ」はスナップショットであり状態ではない
06-25 の達成は7日で陳腐化した。ガバナンスは「達成」ではなく「維持機構」。維持機構は (a) 強制実行（cron）、(b) 正直な集計（問題を集計から漏らさない）、(c) 変化の自動編入（リスト自動発見）の3点セットで初めて成立する。

### 2. 監査スクリプト自身を監査せよ
「ドキュメントが約束する挙動が実装されているか」は別種の乖離。review_after 警告は文書上存在したが実装されていなかった。監査日は再実行しても 06-25 と表示された。**検査機構は検査されないまま信頼される**傾向があり、敵対的レビューの最優先対象にすべき。

### 3. 失敗経路が偽適合に化ける設計を許すな
`ls_remote_branches()` が失敗時に `[]` を返す設計は「ネットワーク断＝ブランチなし＝適合」を意味していた。失敗（unknown）と適合（clean pass）は型レベルで区別する（None vs []）。

### 4. allowlist 監査は腐る。denylist（除外だけ明示）に反転させる
「監査するリポを列挙する」は列挙者の不在で腐った。「全リポを監査し、除外だけを EXCLUSIONS.md で承認付き管理する」構造なら、新リポ・private リポの見落としが構造的に起きない。

### 5. public リポで private 資産を監査する時は出力経路を全部洗う
ダッシュボード・Issue 本文・EXCLUSIONS エントリ・raw JSON、それぞれが実名の漏洩経路になる。マスクは表示層（summary）で一元化し、raw JSON はコミットしない運用にした。

### 6. ブランチ放置の真因は「Web版セッションの完了定義」にある（未解決）
branch-cleanup スキルと CLAUDE.md トリガーを全リポに配ってもなお、06-24〜06-29 の Web版セッションはブランチを残して完了した。配布物による自主起動には限界がある。月次 cron は「検知の保証」であり「発生の防止」ではない。防止策（例: Web版セッション開始プロンプトの定型化）は今後の課題。

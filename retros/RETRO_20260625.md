## 教訓（次セッションへ）

### 1. 機械監査は人の判断より厳しいことを知るべき
人間が「だいたいできている」と思っていても、12軸で機械検査すると2-3件は見落としがある。今回も NASDAQ_backtest の identical なブランチや shopping_product_search の進行中ブランチが、人の目では気付かれない位置にあった。

### 2. CDN キャッシュは監査の天敵
raw.githubusercontent.com は数分〜数十分の遅延がある。「直前にコミットしたものを raw 経由で監査する」のは常に間違いの元。**Contents API + Cache-Control: no-cache + token 認証** がデフォルトであるべき。

### 3. EXCLUSIONS は「逃げ道」ではなく「設計の尊重」
最初は「happiness-system も標準化したほうが綺麗」と考えたが、実態を見ると **設計が成功している段階で活発に使用中**。機械監査の「綺麗さ」のために独自設計を壊すのは本末転倒。EXCLUSIONS は「ルールを破る逃げ道」ではなく「意図的設計の保護機構」。

### 4. ガバナンス整備の「3層」が見えた
- **層1: 文書化**（ルールを書く） → 2026-06-23 にやった
- **層2: 配布**（各リポに反映） → 2026-06-24 にやった
- **層3: 監査**（守られているか確認） → 2026-06-25 にやった（今回）
- **層4: 継続**（月次自動監査・ドリフト検知） → 次のフェーズ

層3まで来てようやくガバナンスが機能し始める。層4は `/schedule` か GitHub Actions で組める。

### 5. shopping_product_search の進行中ブランチを発見した意義
監査の副次効果として、ユーザーの別セッションで進行中の作業（`claude/toddler-undershirt-heat-rash-u9e1mu`）を検知できた。固有コミット有り＝完了前に放置すると本リポのルール違反になるため、次の整備セッション開始時に「あれは完了しましたか？」と確認するきっかけになる。

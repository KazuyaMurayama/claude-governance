# ルール遵守率改善 — 問題分析と実装（rule-enforcement パッケージ）

作成日: 2026-07-02
最終更新日: 2026-07-02
作者: 男座員也（Kazuya Oza）セッション（Claude Code 実装）

## 0. 課題

Web版・VSCode版の両 Claude Code で、CLAUDE.md 等に記載済みのルール・ポリシーが守られずに回答されるケースが散発。指摘すると「書いてあったが見逃した」という応答になる。**書いてあるのに守られない**構造要因を特定し、遵守率を構造的に引き上げる。

## 1. 問題分析（なぜ「書いてあるのに守られない」のか）

実測に基づく4つの構造要因：

### (1) 注意の希釈（ルールが多すぎ・遠すぎる）
各リポの CLAUDE.md は実測 195〜324行・23〜39セクション。グローバル CLAUDE.md はさらに長く、スキル起動表だけで50超のトリガーを列挙している。LLM がセッション全体で「常時アクティブに保持」できるルール数には限りがあり、フラットに並んだ数十のルールは1本あたりの遵守確率が下がる。**ルールを増やすほど1本あたりの遵守率は下がる**というトレードオフが放置されていた。

### (2) タイミングのミスマッチ（読む時点と使う時点のずれ）
CLAUDE.md はセッション開始時に読まれるが、違反が起きるのは**セッション終盤**（成果物報告・ブランチ集約・完了宣言）や**ツール実行の瞬間**（保存先選択）。長いセッションでは開始時に読んだルールが会話の彼方に流れ、自動コンパクションでさらに希薄化する。「守るべき瞬間」にルールが目の前にない。

### (3) 強制力の欠如（すべてが助言テキスト）
ルールは全て散文であり、破っても何も起きない。実証: branch-cleanup トリガーを全43リポに配布済みだったにも関わらず、2026-06-24〜29 の Web版セッションはブランチ4本（固有コミット10件）を放置して完了した。**配布された助言テキストだけでは再発を防げない**ことが監査で確認済み。

### (4) 検証ゲートの不在
「完了と言う前に検証する」ルールはあるが、完了宣言の直前に機械的なチェックポイントが無い。sp-verification-before-completion スキルは存在するが、50超のトリガー表の中の1行であり起動率が低い（(1)の帰結）。

## 2. 対応（実装済み）

3層の防御を、ガバナンス対象の全リポ＋グローバルに展開した。

### 層1: ⛔絶対ルール Top10 ブロック（注意の希釈への対策）
- 正典: `templates/hard-rules-block.md`（`<!-- HARD_RULES_START v1 -->` マーカー管理）
- 違反実績・重要度で選抜した10項目だけを、**CLAUDE.md の最上部（H1直後）**に命令形で配置。「全ルール平等」をやめ、注意を集中させる。
- 展開先: ガバナンス対象42リポ ＋ claude-governance 自身 ＋ グローバル `~/.claude/CLAUDE.md`（`global/CLAUDE.md` 同期済）＋ `templates/repo-CLAUDE.md.template`
- Web版対応: リポ内ファイルなので Web版でも必ず読まれる。

### 層2: 決定論的 hooks（強制力・タイミングの対策）
機械判定可能なルールは散文でなくコードで守る。`.claude/hooks/` に3スクリプト（fail-open 設計＝異常時は必ず素通しでセッションを壊さない）:

| Hook | イベント | 何を守るか |
|---|---|---|
| `pre_write_guard.py` | PreToolUse (Write\|Edit) | ルール3: Desktop への生成を**その場で拒否**（`Desktop\repos\`・`Desktop\投資・不動産\` は許可） |
| `post_bash_guard.py` | PostToolUse (Bash\|PowerShell) | ルール2: `git push` 検出の瞬間に「3列表＋URL検証＋mainのみ」を**リマインド注入** |
| `session_guard.py` | Stop | ルール1/7: **回答終了の瞬間**に非mainブランチ・未pushコミットを検査し、違反なら終了を1回ブロックして是正内容をフィードバック（`stop_hook_active` で2回目は素通し＝無限ループ不可） |

- 配線: 各リポ `.claude/settings.json` に**マージ**（既存の permissions・既存 hooks は全て保持。oogiri の permissions 11件・deep-research の Mermaid 検査 hook で保全を実証）
- ローカル側: `~/.claude/hooks/` ＋ `~/.claude/settings.json`（絶対パス指定）。リポ側 hook が存在する場合はグローバル側が自動で譲る重複排除ロジック内蔵。
- ユニットテスト済み: 違反検出・許可パス・壊れた入力（fail-open）・stop_hook_active・非gitディレクトリの全ケース。

### 層3: 監査による維持（ドリフト検知）
- 監査に **HR**（Top10ブロック存在）と **EH**（hooks 配線＋スクリプト実在）の2軸を追加。
- 月次 GitHub Actions 監査が今後の欠落・剥がれを自動検知（FAIL時 Issue 起票）。
- 正典を変えたら `audits/deploy_rule_enforcement.py` の再実行で全リポへ冪等に再展開できる（マーカー比較で最新版へ自動更新）。

### 展開結果（検証済み）
- **42リポ deploy 成功・失敗ゼロ**（skip: happiness-system=EXCLUSIONS / Soulful-Content・tax-advisor=未ガバナンス / sharetask=CLAUDE.md無し）
- 再監査: 展開済みリポは HR/EH 含め**全軸適合**。ローカル同期監査も IN SYNC。

## 3. 期待効果

| 違反類型（実績あり） | 対策 | 期待される変化 |
|---|---|---|
| ブランチ放置・未push完了 | Top10 #1 ＋ Stop hook | 終了の瞬間に機械検査が入るため、**見逃しても最後に必ず1回捕捉**される |
| 成果物3列表・URL検証漏れ | Top10 #2 ＋ push直後リマインド | ルールが「使う瞬間」に再注入される |
| Desktop への一時ファイル生成 | Top10 #3 ＋ PreToolUse 拒否 | 散文でなく**決定論的にブロック**（遵守率はモデル非依存で100%） |
| 検証なし完了宣言 | Top10 #7 ＋ Stop hook の未push検査 | 部分的に機械化、残りは最上部ルールで注意集中 |
| 命名規則・表記名ミス | Top10 #4/#5（最上部化） | 位置効果による改善（機械化は今後の拡張候補） |

構造上のポイント: **散文ルールの遵守は確率的だが、hooks は決定論的**。最頻出の違反3類型（ブランチ・報告・保存先）に機械ガードが入ったため、「見逃しました」で素通りするパスが閉じた。

## 4. 留意点

1. **Web版での hooks 発火は要実地確認**: リポの `.claude/settings.json` hooks は Claude Code 共通機構だが、Web版サンドボックスでの発火は次回 Web セッションで確認すること（確認方法: Web版で `git checkout -b test` → 応答終了時に session_guard の完了前チェックが出るか）。deep-research には従来からリポ hooks が存在しており、機構自体は利用実績がある。層1（Top10ブロック）は hooks と独立に Web版でも確実に効く。
2. **Stop hook は「1回だけ」ブロック**: 2回目の終了は素通し（無限ループ防止）。意図的にブランチ作業を継続する場合は、その理由を回答に書けばそのまま終了できる。
3. **push リマインドは毎 push 後に出る**: 多数 push するセッションではややノイズになる。実害が出たら「セッション初回のみ」に絞る改修が可能。
4. **hooks は fail-open**: python 不在・cwd 不一致等では黙って素通しする（安全側）。強制力が要る Desktop ガードも同様なので、100%の防御ではなく「ほぼ常に効く網」。
5. **ルール総量は増えている**: Top10 は「追加」であり既存本文の削減はしていない。次の改善候補は、各リポ CLAUDE.md 本文の重複記述を Top10 参照へ置換して総量を減らすこと（今回は変更リスクを考慮し見送り）。
6. **settings.json のローカル hooks はセッション再起動後に有効化**（本セッション自体には効かない）。
